## ADDED Requirements

This document details *changes* to requirements and is additive to the `design.md` document, not a replacement.

### Requirement: ADDED Centralized Secure Suffix Generation

The system SHALL provide a centralized utility for generating unique, cryptographically secure suffixes for E2E test resources.

#### Scenario: Generate a secure suffix

- **Given** The `getUniqueSuffix` function in `src/e2e/helpers/utils.ts`
- **When** The function is called
- **Then** It returns a string containing a timestamp and an 8-character random hex/hex-like string (from a UUID)
- **Then** Subsequent calls within the same millisecond still return unique values due to the random portion

## MODIFIED Requirements

### Requirement: MODIFIED Unique Resource Naming in Test Helpers

The E2E test helpers (Auth, Cookbooks, Recipes) SHALL use the centralized `getUniqueSuffix` utility instead of `Math.random()`.

#### Scenario: Create a unique recipe name

- **Given** A call to `getUniqueRecipeName("My Recipe")`
- **When** The function executes
- **Then** It returns a string in the format "My Recipe <timestamp><random-8-chars>"
- **Then** The `Math.random()` logic is no longer present in `src/e2e/helpers/recipes.ts`

#### Scenario: Create a unique user registration

- **Given** A call to `registerAndLogin` with default options
- **When** The function generates a username and email
- **Then** Both username and email use the suffix generated by `getUniqueSuffix`
- **Then** The `Math.random()` logic is no longer present in `src/e2e/helpers/auth.ts`

## REMOVED Requirements

### Requirement: REMOVED Insecure Randomness in E2E Helpers

Reason for removal: `Math.random()` is not cryptographically secure and is prone to collisions in parallel test runs. It is being replaced by `node:crypto` based generation.

## Traceability

- Proposal element: Use `node:crypto` instead of `Math.random()` -> Requirement: MODIFIED Unique Resource Naming in Test Helpers
- Design decision: Centralized Utility Helper -> Requirement: ADDED Centralized Secure Suffix Generation
- Design decision: Suffix Generation Strategy -> Requirement: ADDED Centralized Secure Suffix Generation

## Non-Functional Acceptance Criteria

#### Scenario: Avoid insecure random number generation

- **Given** The `src/e2e/helpers/` directory
- **When** Scanned for `Math.random()`

### Requirement: Reliability

#### Scenario: Parallel test execution uniqueness

- **Given** Multiple E2E tests running in parallel
- **When** They each call `getUniqueSuffix`
- **Then** Each test receives a unique identifier, preventing resource naming collisions

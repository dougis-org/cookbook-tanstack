## Goal

Ensure that CI commits generated by auto-updating `package-lock.json` in a PR branch are treated as normal commits and run the full CI pipeline, rather than being excluded from checks via `[skip ci]`.

## Acceptance Criteria

1. Given a PR where `package-lock.json` is out of sync and `build-and-test` auto-corrects it:
   - The workflow commits the fix with the message `chore: update package-lock.json` (no `[skip ci]`).
   - The workflow sets an explicit run marker (e.g., environment variable `CI_UPDATED_LOCKFILE=true`) in the triggered run, so downstream audit logic can identify automation-triggered runs.
   - The workflow pushes the update and triggers a second CI run on the same PR branch.
   - The second run executes the full build/test suite.

2. Given the second run finds no additional lockfile drift:
   - No commit is made.
   - The suite runs once and finishes successfully.

3. Pipeline remains idempotent and non-blocking for regular PRs that do not modify `package-lock.json`.

3. Pipeline remains idempotent and non-blocking for regular PRs that do not modify `package-lock.json`.

## Metrics

- Exploration and fix for issue #172.
- CI rerun counts should increment by exactly 1 in drift correction cases (expected behavior).

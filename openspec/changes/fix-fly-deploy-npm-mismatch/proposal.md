## GitHub Issues

- #406

## Why

- **Problem statement:** Fly.io deployments have been failing on every PR merge since ~April 27 because `npm ci` inside the Docker build (Node 24 / npm 11) rejects the committed `package-lock.json`, which was generated by the CI pipeline running Node 22 / npm 10.
- **Why now:** The issue blocks every production deploy. All merges to main since April 27 have failed to ship. The fix is well-understood and low-risk.
- **Business/user impact:** No code has reached production since April 27. Every merged PR is silently stuck.

## Problem Space

- **Current behavior:** The `build-and-test.yml` CI pipeline uses Node 22 (npm 10) to run `npm install` and auto-commit an updated lockfile. The `deploy.yml` pipeline runs `npm ci` inside a `node:24-alpine` Docker image (npm 11). npm 11 enforces stricter peer dependency hoisting than npm 10: it requires `@tanstack/query-core` to appear as a top-level entry in the lockfile to satisfy the `>=5.90.0` peer dep declared by `@tanstack/react-router-ssr-query` and `@tanstack/router-ssr-query-core`. npm 10 nested it under `node_modules/@tanstack/react-query/node_modules/` without a top-level entry, which npm 11 considers invalid. The auto-fix CI "heals" the lockfile using npm 10, so it looks correct but still breaks in Docker.
- **Desired behavior:** The lockfile committed to the repo is generated by the same npm version used in Docker. `npm ci` succeeds in Docker on every deploy. Lockfile drift is caught early in CI (on the PR), not at deploy time.
- **Constraints:** Must not break the existing auto-lockfile-update CI mechanism. Must not regress the existing build, test, or E2E pipeline. The Dockerfile `npm ci` approach is intentional and should be preserved for reproducibility.
- **Assumptions:** Node 24 LTS is stable enough to use in CI. Upgrading `@tanstack/react-query` from `5.96.2` to `5.100.6` is non-breaking (same major, same API).
- **Edge cases considered:** The lockfile auto-commit step pushes back to the PR branch; after the Node version change, npm 11 will generate a lockfile that Docker also accepts. Fork PRs do not push back (no token), so they may surface lockfile issues as a build failure rather than an auto-fix — this is acceptable and existing behavior.

## Scope

### In Scope

- Upgrade `build-and-test.yml` `node-version` from `'22'` to `'24'`
- Upgrade `@tanstack/react-query` to `5.100.6` in `package.json`
- Align all `@tanstack/react-router`, `@tanstack/react-start`, `@tanstack/react-router-ssr-query`, `@tanstack/router-plugin`, and related packages to a consistent, matching minor version set
- Pin `@trpc/client`, `@trpc/server`, and `@trpc/tanstack-react-query` to exact versions (remove `^`)
- Regenerate `package-lock.json` locally using Node 24 / npm 11 after package.json changes
- Verify `npm ci` passes in Docker after the lockfile is regenerated
- Close GitHub issue #406

### Out of Scope

- Changing the `node:24-alpine` base image in the Dockerfile
- Upgrading any unrelated dependencies
- Changing the deploy trigger (still fires on merged PRs to main)
- Adding new CI workflows or jobs beyond the node-version bump

## What Changes

- `package.json` — version bumps for `@tanstack/react-query`, aligned TanStack router/start family, exact-pinned tRPC packages
- `package-lock.json` — fully regenerated with npm 11 (Node 24); includes top-level `@tanstack/query-core` entry
- `.github/workflows/build-and-test.yml` — `node-version: '22'` → `node-version: '24'`

## Risks

- **Risk:** Upgrading `@tanstack/react-query` to `5.100.6` introduces a breaking API change.
  - **Impact:** Runtime errors in query-dependent components.
  - **Mitigation:** The upgrade is within the same major (5.x). TanStack Query follows semver. Full test suite (unit + E2E) runs after the change and must pass before merge.

- **Risk:** Aligning TanStack router/start packages to a newer minor exposes a regression.
  - **Impact:** Routing or SSR breakage.
  - **Mitigation:** All existing E2E tests cover the routing layer. CI must pass before merge.

- **Risk:** npm 11 on CI generates a lockfile that diverges from what developers have locally (if they are still on npm 10 / Node 22).
  - **Impact:** Local `npm ci` may fail for developers on older Node.
  - **Mitigation:** The `.nvmrc` or Node engine field in `package.json` should document Node 24 as the required runtime. Developers can run `npm install` locally to sync.
